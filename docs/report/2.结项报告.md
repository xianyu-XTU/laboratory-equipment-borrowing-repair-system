## 一、项目概述

### 1. 项目背景

随着高校实验教学活动日益丰富，实验室设备种类和使用频率不断增加。传统实验室设备管理方式主要依赖纸质登记、Excel 表格或口头沟通，存在信息不集中、设备去向不清、审批流程不规范、归还不及时、损坏责任难追踪、维修记录不完整等问题。

为提高实验室设备资源利用率和管理规范性，本项目开发了面向学生、实验员和管理员的校园实验室设备管理系统。系统围绕设备信息维护、设备借用申请、借用审批、借出归还、设备报修、维修处理、公告发布和数据统计等核心业务，实现实验室设备管理流程线上化、信息化和可追溯化。

### 2. 项目目标

本项目最终目标是构建一个功能完整、界面清晰、流程闭环、可运行演示的实验室设备管理系统，具体目标如下：

- 建立统一的实验室设备信息管理平台，实现设备基础信息集中维护。
- 实现设备借用申请、审批、借出和归还流程线上化。
- 实现设备状态动态更新，支持正常、待审批、借用中、维修中等状态管理。
- 实现设备报修申请、维修处理和维修记录管理。
- 支持管理员发布系统公告，方便通知实验室相关事项。
- 支持多角色登录与权限控制，区分学生、实验员和管理员操作范围。
- 实现基础统计分析功能，辅助管理员了解设备使用和维修情况。
- 完成系统部署、功能测试、文档整理和项目结项材料归档。

## 二、需求分析

### 1. 功能需求

系统面向三类核心用户：学生、实验员和管理员。不同角色登录后进入不同页面，并只能访问其权限范围内的功能。

#### （1）学生端需求

学生用户主要完成设备查看、借用申请、个人借用记录查询和设备报修等操作：

- 登录系统并进入学生端面板。
- 查看设备列表、设备状态和设备分类。
- 对可借设备提交借用申请。
- 查看个人借用申请审批状态。
- 查看个人借用记录和归还状态。
- 对自己正在借用的设备提交报修申请。
- 查看个人报修记录及处理进度。
- 查看系统公告和实验室通知。

#### （2）实验员端需求

实验员用户主要负责设备日常管理、借用审批、归还登记和报修处理：

- 登录系统并进入实验员端面板。
- 查看设备列表、设备分类和设备当前状态。
- 新增或修改设备信息。
- 查看学生提交的借用申请。
- 审批借用申请。
- 查看借用记录并登记设备归还。
- 查看报修记录并处理维修状态。
- 查看系统公告和基础统计信息。

#### （3）管理员端需求

管理员用户负责系统整体维护、用户管理、设备管理和数据统计：

- 登录系统并进入管理员端面板。
- 查看、新增、修改用户信息，调整用户状态。
- 管理设备分类和设备基础信息。
- 新增、修改、删除设备信息。
- 查看并审批借用申请。
- 查看借用记录、报修记录和维修状态。
- 发布系统公告。
- 查看系统统计概览。

#### （1）登录与权限流程

用户在登录页输入账号和密码后，前端调用后端登录接口。登录成功后，后端返回 token 和用户基本信息，前端将 token 与用户信息保存到 LocalStorage。后续请求统一携带 `Authorization: Bearer <token>` 请求头。前端根据用户角色跳转到不同页面：管理员进入 `board_admin.html`，实验员进入 `board_lab.html`，学生进入 `board_student.html`。

```text
用户输入账号密码
      │
      ▼
调用 /api/auth/login
      │
      ▼
后端校验账号、密码和用户状态
      │
      ▼
返回 token、userId、username、realName、roleCode
      │
      ▼
前端保存登录信息并按角色跳转页面
```

#### （2）设备借用流程

学生登录后查看设备列表，选择可借设备并提交借用申请。管理员或实验员审批申请。审批通过后，系统生成借用记录并将设备状态改为已借出；审批拒绝则记录审批结果，流程结束。设备使用完成后，由实验员或管理员登记归还，系统更新借用记录并将设备状态恢复为可借。

```text
学生登录
   │
   ▼
查看可借设备
   │
   ▼
提交借用申请
   │
   ▼
管理员 / 实验员审批
   │
   ├── 拒绝：记录审批结果，流程结束
   │
   └── 通过：生成借用记录，设备状态改为已借出
                 │
                 ▼
              设备使用
                 │
                 ▼
          实验员 / 管理员登记归还
                 │
                 ▼
          更新借用记录和设备状态
```

#### （3）设备报修流程

用户发现设备异常后可提交报修申请。系统记录设备、报修人和故障描述，并将设备状态改为维修中。实验员或管理员查看报修记录后进行处理，可将报修状态更新为维修中或已完成。维修完成后，系统根据设备是否仍存在借用中记录，将设备状态恢复为已借出或可借。

```text
发现设备故障
   │
   ▼
提交报修申请
   │
   ▼
系统记录故障并更新设备状态为维修中
   │
   ▼
实验员 / 管理员处理报修
   │
   ▼
填写处理说明并更新维修状态
   │
   ▼
维修完成后恢复设备状态
```

## 三、系统设计

### 1. 系统架构设计

系统采用前后端分离思想。前端由静态 HTML 页面和 JavaScript 组成，负责页面展示、表单交互和接口请求；后端基于 Spring Boot 构建，负责认证、权限校验、业务逻辑处理和数据返回；数据库使用 MySQL 保存角色、用户、设备、分类、借用申请、借用记录、报修记录和公告数据。

```text
用户浏览器
   │
   │  HTML 页面 / JavaScript 交互
   ▼
前端 frontend
   │
   │  HTTP JSON 请求，携带 Bearer Token
   ▼
后端 backend：Spring Boot
   │
   ├── Controller：接收请求，返回统一结果
   ├── Service：处理认证、借用、审批、归还、报修等业务
   ├── Mapper：通过 MyBatis-Plus 访问数据库
   └── Entity / DTO：封装数据表实体和请求响应对象
   │
   ▼
MySQL 数据库 lab_equipment
```

### 2. 模块设计

系统主要包括用户与权限管理、设备信息管理、借用申请与审批、报修与维修管理和公告管理五个核心模块。各模块之间通过用户身份、设备状态和业务记录相互关联，共同构成完整的实验室设备管理流程。

#### （1）用户与权限管理模块

该模块负责用户注册登录、角色区分、个人信息维护和密码修改。系统通过用户角色字段区分学生、实验员和管理员，不同角色登录后进入对应功能页面，并只能访问其权限范围内的接口。

![设备信息管理模块流程图](./assets/设备信息管理模块流程图.png)

![借用流程（二步式审批流程）图](./assets/借用流程（二步式审批流程）.png)

## 四、系统实现  

### 1. 关键技术

本系统采用前后端分离的设计思想，前端负责页面展示和用户交互，后端负责业务处理、权限校验和数据访问，数据库负责保存用户、设备、借用、报修和公告等业务数据。系统在结项阶段已完成主要功能开发，核心技术和实现要点如下。

#### （1）Spring Boot 后端框架

后端基于 Spring Boot 3 搭建，使用 Spring MVC 提供 RESTful 接口。系统按照 Controller、Service、Mapper、Entity、DTO 等层次组织代码，降低各模块之间的耦合度，便于后期维护和扩展。

Controller 层负责接收前端请求并返回统一结果；Service 层负责实现登录、借用、审批、归还、报修等核心业务逻辑；Mapper 层基于 MyBatis-Plus 完成数据库访问；Entity 和 DTO 分别用于映射数据库表和封装接口请求、响应数据。

#### （2）MyBatis-Plus 数据访问技术

系统使用 MyBatis-Plus 简化数据库增删改查操作。对于用户、设备、分类、借用申请、借用记录、报修记录和公告等表，均建立了对应实体类和 Mapper 接口。

MyBatis-Plus 提供的通用 Mapper 方法减少了重复 SQL 编写工作，同时系统使用分页查询支持设备列表、用户列表、借用记录、报修记录和公告列表展示，提高了后台接口的可维护性和查询效率。

#### （3）MySQL 数据库存储

系统数据库名称为 `lab_equipment`，核心表包括：

```text
role
user
device_category
device
borrow_apply
borrow_record
repair_record
notice
```

其中，`borrow_apply` 用于保存设备借用申请，`borrow_record` 用于保存实际借用记录。这样可以清楚地区分“申请审批阶段”和“实际借出阶段”。`repair_record` 用于保存设备报修和维修处理信息，方便后续查询和统计。

#### （4）Token 登录认证与角色权限控制

系统登录成功后，后端返回 token、用户 ID、用户名、真实姓名和角色编码。前端将 token 与用户信息保存到 LocalStorage，后续请求统一在请求头中携带：

```text
Authorization: Bearer <token>
```

后端通过拦截器解析 token 并保存当前登录用户信息。对于需要权限限制的接口，系统使用自定义 `@RequireRole` 注解进行控制。例如：用户管理接口仅管理员可访问；借用审批、归还和报修处理接口允许管理员和实验员访问；学生只能提交借用申请和查看自己的相关记录。

#### （5）前端统一 API 封装

前端使用 HTML、CSS 和 JavaScript 实现。所有后端请求统一封装在 `frontend/js/api.js` 中，便于集中管理接口地址、请求头、token 携带、异常处理和角色跳转逻辑。

当前前端根据登录返回的角色编码进行页面跳转：

| 角色编码 | 页面 |
|---|---|
| `ADMIN` | `board_admin.html` |
| `LAB_ADMIN` | `board_lab.html` |
| `STUDENT` | `board_student.html` |

这种方式使三个角色的页面职责更加清晰，也便于在前端进行功能区分。

#### （6）设备状态联动处理

设备状态是系统业务的核心之一。系统通过状态字段表示设备当前情况：

| 状态值 | 含义 |
|---|---|
| 1 | 可借 |
| 2 | 已借出 |
| 3 | 维修中 |
| 4 | 停用 / 报废 |

当学生提交借用申请时，系统会检查设备是否处于可借状态；当申请审批通过后，系统生成借用记录并将设备状态修改为已借出；当设备归还后，设备状态恢复为可借；当设备报修后，设备状态修改为维修中；维修完成后，系统根据设备是否仍存在借用中记录决定恢复为已借出或可借。

#### （7）统一返回结果与异常提示

系统后端使用统一返回结果类 `Result` 封装接口响应，返回内容一般包括状态码、提示信息和数据内容。前端收到响应后可统一判断请求是否成功，并给用户显示对应提示。

统一返回结构提高了前后端接口的一致性，也减少了前端对不同接口返回格式的适配工作。

### 3. 核心代码片段

本节展示系统中具有代表性的关键代码，包括前端请求封装、登录接口、权限注解、设备借用审批和报修处理等内容。

#### （1）前端统一请求封装

前端在 `frontend/js/api.js` 中封装统一请求方法。该方法会自动读取本地保存的 token，并放入请求头中。如果后端返回 401，说明登录失效，前端会清除本地登录信息并跳转到登录页面。

```javascript
const API_BASE = "http://localhost:8080";

async function request(path, options = {}) {
  const token = localStorage.getItem("token");
  const headers = {
    "Content-Type": "application/json",
    ...(options.headers || {})
  };

  if (token) {
    headers["Authorization"] = "Bearer " + token;
  }

  const res = await fetch(API_BASE + path, {
    ...options,
    headers
  });

  const json = await res.json();

  if (res.status === 401) {
    localStorage.removeItem("token");
    localStorage.removeItem("loginUser");
    window.location.href = "login.html";
  }

  return json;
}
```

该代码实现了前端接口请求的统一入口，避免每个页面重复编写 token 携带和异常处理逻辑。

#### （2）用户登录接口

后端登录接口位于认证控制器中，前端提交账号和密码后，后端调用业务层完成账号校验和登录信息生成。

```java
@RestController
@RequestMapping("/api/auth")
public class AuthController {

    private final AuthService authService;

    public AuthController(AuthService authService) {
        this.authService = authService;
    }

    @PostMapping("/login")
    public Result<LoginResponse> login(@RequestBody LoginRequest request) {
        return Result.ok(authService.login(request));
    }

    @PostMapping("/register")
    public Result<Void> register(@RequestBody RegisterRequest request) {
        authService.register(request);
        return Result.ok();
    }
}
```

登录成功后，系统返回 token、用户 ID、用户名、真实姓名和角色编码，前端根据角色编码跳转到对应页面。

#### （3）角色权限注解

系统使用自定义 `@RequireRole` 注解标记需要特定角色才能访问的接口。该注解可作用在类或方法上，用于声明允许访问的角色列表。

```java
@Target({ElementType.METHOD, ElementType.TYPE})
@Retention(RetentionPolicy.RUNTIME)
public @interface RequireRole {
    String[] value();
}
```

例如，用户管理接口只允许管理员访问：

```java
@RestController
@RequestMapping("/api/users")
@RequireRole({"ADMIN"})
public class UserController {
    // 用户管理相关接口
}
```

借用审批和归还接口允许管理员和实验员访问：

```java
@RequireRole({"LAB_ADMIN", "ADMIN"})
@PostMapping("/approve")
public Result<Void> approve(@RequestBody BorrowApproveRequest request) {

```

通过这种方式，系统实现了接口级权限控制，避免普通学生访问管理功能。

#### （4）设备借用审批核心逻辑

借用审批是系统的核心业务之一。审批通过时，系统需要修改申请状态、生成借用记录，并同步更新设备状态为已借出；审批拒绝时，只记录审批结果。

```java
@Transactional
public void approve(BorrowApproveRequest request) {
    BorrowApply apply = borrowApplyMapper.selectById(request.getApplyId());
    if (apply == null) {
        throw new RuntimeException("借用申请不存在");
    }
    if (!Objects.equals(apply.getStatus(), 0)) {
        throw new RuntimeException("该申请已处理");
    }

    Device device = deviceMapper.selectById(apply.getDeviceId());
    if (device == null) {
        throw new RuntimeException("设备不存在");
    }

    apply.setStatus(request.getApproved() ? 1 : 2);
    apply.setApproveRemark(request.getRemark());
    apply.setApproveTime(LocalDateTime.now());
    apply.setApproverId(AuthContext.getUserId());
    borrowApplyMapper.updateById(apply);

    if (request.getApproved()) {
        if (!Objects.equals(device.getStatus(), 1)) {
            throw new RuntimeException("设备当前不可借");
        }

        BorrowRecord record = new BorrowRecord();
        record.setApplyId(apply.getId());
        record.setUserId(apply.getUserId());
        record.setDeviceId(apply.getDeviceId());
        record.setBorrowTime(LocalDateTime.now());
        record.setStatus(1);
        borrowRecordMapper.insert(record);

        device.setStatus(2);
        deviceMapper.updateById(device);
    }
}
```

该方法使用事务控制，保证借用申请、借用记录和设备状态三部分数据同时成功或同时回滚，避免出现审批成功但设备状态未更新的问题。

#### （5）设备归还核心逻辑

设备归还是借用流程的结束环节。归还时，系统需要更新借用记录状态和归还时间，并将设备状态恢复为可借。

```java
@Transactional
public void returnDevice(Long recordId) {
    BorrowRecord record = borrowRecordMapper.selectById(recordId);
    if (record == null) {
        throw new RuntimeException("借用记录不存在");
    }
    if (!Objects.equals(record.getStatus(), 1)) {
        throw new RuntimeException("该记录不是借用中状态");
    }

    record.setReturnTime(LocalDateTime.now());
    record.setStatus(2);
    borrowRecordMapper.updateById(record);

    Device device = deviceMapper.selectById(record.getDeviceId());
    if (device != null && Objects.equals(device.getStatus(), 2)) {
        device.setStatus(1);
        deviceMapper.updateById(device);
    }
}
```

该代码保证设备归还后，借用记录和设备状态保持一致。

#### （6）设备报修核心逻辑

报修功能用于处理设备故障。学生报修时，系统会校验设备是否存在，并限制学生只能报修自己正在借用的设备。报修提交后，设备状态会被改为维修中。

```java
@Transactional
public void report(RepairReportRequest request) {
    Device device = deviceMapper.selectById(request.getDeviceId());
    if (device == null) {
        throw new RuntimeException("设备不存在");
    }
    if (Objects.equals(device.getStatus(), 4)) {
        throw new RuntimeException("停用设备不能报修");
    }

    if ("STUDENT".equals(AuthContext.getRoleCode())) {
        LambdaQueryWrapper<BorrowRecord> wrapper = new LambdaQueryWrapper<>();
        wrapper.eq(BorrowRecord::getUserId, AuthContext.getUserId())
               .eq(BorrowRecord::getDeviceId, request.getDeviceId())
               .eq(BorrowRecord::getStatus, 1);

        Long count = borrowRecordMapper.selectCount(wrapper);
        if (count == null || count == 0) {
            throw new RuntimeException("学生只能报修自己正在借用的设备");
        }
    }

    RepairRecord record = new RepairRecord();
    record.setDeviceId(request.getDeviceId());
    record.setUserId(AuthContext.getUserId());
    record.setDescription(request.getDescription());
    record.setStatus(0);
    record.setCreateTime(LocalDateTime.now());
    repairRecordMapper.insert(record);

    device.setStatus(3);
    deviceMapper.updateById(device);
}
```

该代码体现了系统对设备报修权限和设备状态联动的控制，避免学生随意报修未借用设备。

## 六、用户手册

### 1. 环境准备

运行系统前需准备以下环境：

- JDK 17 及以上版本。
- Maven 3.6 及以上版本。
- MySQL 8.0。
- Git。
- Chrome / Edge 等主流浏览器。
- IntelliJ IDEA 或 VS Code。

其中，后端依赖 JDK 与 Maven 运行，数据库依赖 MySQL，前端为静态页面，可直接通过浏览器访问。

### 2. 获取项目代码

```bash
git clone https://github.com/xianyu-XTU/lab-equipment-management-system.git
cd lab-equipment-management-system
```

下载完成后，项目主要目录如下：

```text
lab-equipment-management-system
├── backend       后端 Spring Boot 项目
├── frontend      前端静态页面
├── database      数据库脚本
├── docs          项目文档
└── start_all.bat 一键启动脚本
```

### 3. 初始化数据库

首次运行系统前，需要先创建数据库并导入基础数据。

1. 打开 MySQL、Navicat 或其他数据库管理工具。
2. 执行 `database/lab_equipment_create.sql`，创建数据库、表结构和基础测试数据。
3. 如需更多测试数据，可继续执行 `database/lab_equipment_data.sql`。
4. 确认数据库名称为 `lab_equipment`。

基础测试账号如下：

| 角色 | 用户名 | 密码 |
|---|---|---|
| 管理员 | `admin` | `123456` |
| 实验员 | `lab` | `123456` |
| 学生 | `student` | `123456` |

### 4. 修改后端配置

打开后端配置文件：

```text
backend/src/main/resources/application.yml
```

根据本机 MySQL 配置修改数据库连接信息，重点检查数据库地址、用户名和密码是否正确：

```yaml
spring:
  datasource:
    url: jdbc:mysql://localhost:3306/lab_equipment?useUnicode=true&characterEncoding=utf8&serverTimezone=Asia/Shanghai&allowPublicKeyRetrieval=true
    username: root
    password: root
```

如果本机 MySQL 密码不是 `root`，需要改成自己的数据库密码；如果 MySQL 端口不是 `3306`，也需要同步修改连接地址。

### 5. 一键启动系统

项目根目录下提供了 `start_all.bat` 一键启动脚本。数据库初始化和后端配置完成后，可直接双击运行：

```text
start_all.bat
```

也可以在命令行中进入项目根目录后执行：

```bat
start_all.bat
```

脚本会按照项目配置启动系统所需服务。启动完成后，后端默认访问地址为：

```text
http://localhost:8080
```

前端登录页面位于：

```text
frontend/login.html
```

如果脚本中已配置自动打开前端页面，则等待脚本执行完成后即可在浏览器中登录系统；如果没有自动打开，可手动进入 `frontend` 目录并打开 `login.html`。

### 6. 手动启动方式

如果一键启动脚本无法正常运行，也可以采用手动方式启动后端和前端。

#### （1）启动后端服务

```bash
cd backend
mvn spring-boot:run
```

后端默认运行在：

```text
http://localhost:8080
```

#### （2）启动前端页面

进入 `frontend` 目录，直接用浏览器打开：

```text
frontend/login.html
```

也可以使用 VS Code 的 Live Server 插件运行前端页面。

### 7. 基本使用说明

#### （1）学生使用流程

1. 使用学生账号登录。
2. 进入学生端面板。

2. 进入实验员端面板。

4. 审批学生提交的借用申请。
5. 登记设备归还。
6. 查看并处理设备报修。
7. 查看系统公告和统计概览。

1. 使用管理员账号登录。
2. 进入管理员端面板。
3. 管理用户、设备和设备分类。
4. 审批借用申请。
5. 查看借用记录和报修记录。

7. 查看系统统计概览。

### 8. 常见问题说明

| 问题 | 可能原因 | 解决方法 |
|---|---|---|
| 后端启动失败 | JDK 或 Maven 未配置 | 检查 `java -version` 和 `mvn -version` 是否正常 |
| 数据库连接失败 | MySQL 未启动或账号密码错误 | 启动 MySQL，并检查 `application.yml` 配置 |
| 登录失败 | 数据库未初始化或账号密码错误 | 重新执行数据库脚本，使用测试账号登录 |
| 前端请求失败 | 后端未启动或端口不是 8080 | 确认后端已启动，并检查 `frontend/js/api.js` 中的 `API_BASE` |
| 一键脚本无反应 | 脚本路径或运行权限问题 | 在项目根目录使用命令行执行 `start_all.bat` 查看错误信息 |
