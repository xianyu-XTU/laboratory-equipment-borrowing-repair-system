# 校园实验室设备管理系统测试计划

## 1. 测试计划概述

### 1.1 测试目的
本次测试核心目标为验证校园实验室设备管理系统的功能完整性、运行稳定性，确保系统所有业务功能无错误、操作流程顺畅，符合需求规格要求，能够满足学生、实验员、管理员三类角色的日常使用场景，避免因功能异常影响实验室设备管理工作的正常开展。

### 1.2 测试范围
覆盖系统所有核心业务模块，结合系统ER模型及业务流程，具体范围如下：

1. 用户登录与权限模块：账号登录、角色权限控制（学生、实验员、管理员）
2. 设备管理模块：设备分类管理、设备信息增删改查、设备状态维护、设备查询筛选
3. 借用申请与审批模块：借用申请发起、修改/撤销、实验员审批、借出/归还登记、逾期提醒
4. 故障报修与维修模块：报修发起、维修接单、维修状态更新、设备状态联动
5. 公告与数据统计模块：公告发布/查看、设备/借用/报修数据统计
6. 前端交互与页面展示：页面布局、表单校验、按钮交互、页面跳转
7. 数据一致性：业务操作前后数据库数据同步、关联表数据匹配

### 1.4 测试依据
1. 校园实验室设备管理系统需求规格说明书
2. 系统ER模型（用户、设备、借用申请、报修记录等关联关系）
3. 系统业务流程文档（借用、审批、报修等核心流程）
4. 通用软件功能测试规范

## 2. 测试策略与方法

### 2.1 测试策略
本次测试以手工功能测试为主，辅助以接口测试和数据一致性验证，具体策略如下：

1. 按“模块拆分、角色覆盖”的原则，逐一测试每个业务模块的正向、反向、边界场景；
2. 优先测试核心业务流程（借用申请→审批→借出→归还、报修→维修→完成），再测试非核心功能；
3. 发现缺陷后，及时反馈给开发人员，修复后立即执行回归测试，确保缺陷彻底解决且不影响其他功能；
4. 测试过程中重点关注数据一致性，确保前端操作与数据库数据同步无误。

### 2.2 测试方法

| 测试类型 | 测试方法 | 适用场景 |
|---|---|---|
| 功能测试 | 手工测试：按照测试用例步骤，模拟用户操作，验证功能是否符合预期；等价类划分、边界值分析：覆盖正常、异常、边界场景 | 所有业务模块的功能验证，如登录、设备管理、借用审批等 |
| 接口测试 | 使用Apifox工具，调用系统接口，验证接口入参、出参、响应状态码是否正确，接口逻辑是否正常 | 核心接口，如借用申请提交、审批、设备状态更新等 |
| 数据一致性测试 | 手工查询数据库，对比前端操作前后的数据变化，验证关联表数据匹配、状态同步无误 | 所有涉及数据增删改查的操作，如设备新增、借用申请提交、报修状态更新等 |
| 前端交互测试 | 手工操作前端页面，检查布局、按钮、表单、弹窗等交互效果，使用浏览器开发者工具排查JS报错 | 所有前端页面的展示与交互，如登录页、设备列表页、申请表单页等 |

### 2.3 测试工具
1. 测试用例与缺陷管理：Excel（轻量）、禅道（团队协作）
2. 接口测试：Apifox
3. 数据库验证：MySQL Workbench
4. 前端测试：Chrome DevTools、Snipaste（截图记录）

## 3. 测试内容与测试用例设计规范

### 3.1 测试内容细分

#### 3.1.1 用户登录与权限模块
1. 登录功能：正常登录、错误账号/密码登录、账号锁定、验证码（如有）验证；
2. 角色权限：学生、实验员、管理员各自功能菜单可见性、操作权限验证。

#### 3.1.2 设备管理模块
1. 设备分类：新增、编辑、删除分类，分类名称唯一性验证；
2. 设备信息：新增、编辑、删除设备，字段格式校验、设备编号唯一性验证；
3. 设备查询：按编号、名称、状态、分类筛选，分页、导出功能验证；
4. 设备状态：正常、维修、报废、已借出状态的切换与同步验证。

#### 3.1.3 借用申请与审批模块
1. 借用申请：发起申请、修改申请、撤销申请，借用时间合理性、设备状态校验；
2. 审批流程：实验员同意/驳回申请、审批备注、批量审批、审批状态同步；
3. 借还登记：借出登记、归还登记，设备状态联动、逾期归还提示、逾期记录标记；
4. 异常场景：重复申请、设备已借出申请、借用时间不合理等拦截验证。

#### 3.1.4 故障报修与维修模块
1. 报修发起：选择设备、填写故障描述，设备状态校验；
2. 维修处理：接单、维修中、维修完成状态更新，维修结果记录；
3. 状态联动：报修后设备变为“维修中”，维修完成后恢复对应状态。

#### 3.1.5 公告与数据统计模块
1. 公告管理：管理员发布、编辑、删除公告，学生端查看公告；
2. 数据统计：设备、借用、报修数据统计准确性，报表导出功能验证。

#### 3.1.6 数据一致性与前端交互
1. 数据一致性：前端操作后，数据库数据同步更新，关联表数据匹配；
2. 前端交互：页面布局、按钮点击、表单校验、页面跳转、无JS报错。

### 3.2 测试用例设计规范
测试用例需符合以下规范，确保可执行、可追溯，覆盖所有测试场景：

1. 每个用例对应一个具体功能点或场景，明确“前置条件、操作步骤、预期结果”；
2. 覆盖正向场景（正常操作）、反向场景（错误操作、异常输入）、边界场景（参数边界、时间边界）；
3. 每个用例需明确对应需求点，便于后续追溯。

## 4. 缺陷管理规范

### 4.1 缺陷定义
符合以下情况之一，判定为缺陷：

1. 功能未实现或实现与需求不符；
2. 操作后出现错误提示、系统崩溃、页面卡死；
3. 数据录入、查询、修改、删除后，数据不一致；
4. 前端页面布局错乱、按钮无法点击、表单校验失效；
5. 越权操作未被拦截；
6. 核心业务流程无法正常推进。

### 4.2 缺陷提交规范
提交缺陷时，需包含以下信息，确保开发人员快速定位并修复：

1. 缺陷标题：简洁明了，说明“模块+问题”（如：借用申请模块，重复提交申请无拦截）；
2. 前置条件：测试环境、账号、相关数据等；
3. 操作步骤：详细的操作流程，可复现问题；
4. 预期结果与实际结果：明确对比，说明问题所在；
5. 附件：截图。

## 附录：术语定义
1. 正向测试：模拟正常用户操作，验证功能是否符合预期；
2. 反向测试：模拟错误操作、异常输入，验证系统是否能有效拦截；
3. 回归测试：缺陷修复后，重新测试对应功能，确保缺陷解决且无新问题；
4. 核心业务流程：学生借用设备→实验员审批→借出→归还、设备报修→维修→完成。
